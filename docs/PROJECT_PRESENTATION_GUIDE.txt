SnakeBiteRx Project Guide (Beginner + Detailed)

This is detailed guide you can use to study and present this project.

------------------------------------------------------------

One-line idea

SnakeBiteRx is a research prototype that estimates likely venom type from:
* wound photo,
* selected symptoms,
* location (country/state),
* context (time since bite, bite circumstance, age, weight).

It returns a probability distribution, not a medical diagnosis.

------------------------------------------------------------

Safety first (must say this in presentation)

* It is not a medical device.
* It is not FDA/CE approved.
* It can be wrong.
* If confidence is low, it shows unknown.
* In real emergencies, users must follow local emergency care immediately.

------------------------------------------------------------

Main goal of the system

The project tries to answer:

"Given image + symptoms + location + context, what is the probability for each venom pattern class?"

It does not try to identify exact snake species with certainty. Species output is a ranked hint.

------------------------------------------------------------

Output classes (same everywhere)

The 5 output classes are:
1. cytotoxic
2. hemotoxic
3. neurotoxic
4. non_venomous
5. not_snakebite

Important: these same labels are used by image model, symptom model, geo model, and final fusion.

------------------------------------------------------------

Full architecture (simple view)

There are 4 branches:

1. Wound image branch (CNN)
2. Symptom branch (knowledge-base probability matrix)
3. Geography branch (country/state prior from geo data)
4. Context branch (small heuristic prior: time, circumstance, age, weight)

Then all 4 are combined by log-space fusion.

------------------------------------------------------------

What goes IN (inputs)

Required for /predict
* file (image upload)
* country (string)

Optional fields (with defaults)
* state (default empty)
* symptoms as JSON string array (default [])
* time_since_bite_hours (default 3.0)
* bite_circumstance (default "unknown")
* age_years (default 35.0)
* weight_kg (default 60.0)

------------------------------------------------------------

What comes OUT (outputs)

Main outputs from /predict:

* final_probability: 5-class final probability vector
* top_class: raw best class from final vector
* top_confidence: probability value of top_class
* prediction_uncertain: true/false
* display_top_class: class to show in UI (unknown when low confidence)

Also returns branch details:
* wound_probability, symptom_probability, geo_probability
* branch top classes/confidences
* wound_detail (per-backbone info + uncertainty details)
* image_quality (blur check)
* selected_symptoms_ranked
* snake_species_top (ranked hints)
* debug object with fusion internals

------------------------------------------------------------

Wound image model (how + why)

Preferred model file
* models/wound_ensemble.pt

Ensemble members
* EfficientNet-B3
* ResNet50
* DenseNet121

Default ensemble fusion weights
* EfficientNet-B3: 0.58
* ResNet50: 0.26
* DenseNet121: 0.16

Why ensemble?
* Different CNNs make different mistakes.
* Weighted averaging of softmax outputs is more stable than a single model.

Fallback model
* models/wound_mobilenet.pt (single backbone, smaller)

------------------------------------------------------------

Why these models, why not others?

Why EfficientNet-B3 + ResNet50 + DenseNet121?
* They are proven, common CNN families with different feature behaviors.
* Together they reduce single-model bias (lighting, texture, skin tone variation, camera quality).
* They are available in mature PyTorch ecosystems, so loading, testing, and deployment are easier.
* This project uses a practical engineering choice: strong accuracy/stability with manageable complexity.

Why not only one bigger model?
* One model can be overconfident on specific data artifacts.
* A single model failure can directly dominate prediction.
* Ensemble gives error diversity and safer behavior for ambiguous images.

Why not very new architectures (ViT-only, ConvNeXt-only, etc.)?
* They can work, but need retraining, careful benchmarking, and more tuning for this exact dataset.
* This project prioritizes reproducible and stable behavior over chasing newest architecture.
* The current stack already includes uncertainty gates and multimodal fallback, so risk is lower.

Why not object detection/segmentation first?
* Current target is class-level venom pattern estimation, not lesion boundary detection.
* Detection pipelines increase annotation and engineering cost.
* Could be a future extension if you collect reliable wound segmentation labels.

Why symptoms/geo/context are kept (not pure deep learning)?
* Clinical signs may not be visible in one photo.
* Geo and context provide useful prior signals.
* Multimodal design gives graceful fallback when the image branch is uncertain or missing.

------------------------------------------------------------

Wound uncertainty logic

Wound result is marked uncertain (wound_uncertain = true) when:

1. max wound confidence < 0.65, or
2. gap between top1 and top2 < 0.12

So, uncertain means the wound softmax is weak or too flat/ambiguous.

------------------------------------------------------------

Final uncertainty logic

After multimodal fusion:
* if final top confidence < 0.60, then
  - prediction_uncertain = true
  - display_top_class = "unknown"

This is a safety behavior to avoid over-confident labels in UI.

------------------------------------------------------------

Symptom model

Source: data/symptom_data/processed/symptom_dataset.csv

How it works:
* each symptom has weighted links to venom classes,
* selected symptoms are summed,
* normalized to make a probability distribution.

No selected symptoms (or unknown symptom keys) => uniform distribution.

------------------------------------------------------------

Geo model

Sources include:
* data/geo_data/snake_geo_clean.csv
* built assets in models/geo_region_prior.json, models/geo_species_table.json

How it works:
* country/state -> venom prior distribution,
* fallback chain: exact region -> country default -> global default.

Important limitation:
* This is based on occurrence-style geographic data, not direct bite incidence records.

------------------------------------------------------------

Context model (small heuristic prior)

Context lightly nudges probabilities, for example:
* very early hours: slight hemo/cyto bump
* later hours: slight neuro bump
* nocturnal/sleeping/indoor/krait-like context: neuro bump
* child/low-weight: very mild adjustments

It is intentionally weak and not a substitute for exam.

------------------------------------------------------------

Fusion math (core idea)

Fusion is in log space:

w_w*log(p_wound) + w_s*log(p_symptom) + w_g*log(p_geo) + w_c*log(p_context)

Then softmax -> final probabilities.

Why log-space?
* better behavior than naive linear mixing,
* cleaner multiplicative combination of evidences.

------------------------------------------------------------

Fusion weight regimes (very important)

A) No wound model loaded
* wound: 0.38
* symptom: 0.22
* geo: 0.22
* context: 0.18

B) Wound model loaded but uncertain
* wound: 0.28
* symptom: 0.28
* geo: 0.24
* context: 0.20

C) Wound model loaded and confident
* wound: 0.80
* symptom: 0.07
* geo: 0.10
* context: 0.03

Interpretation:
* if image is strong, image dominates;
* if image is weak or missing, symptom/geo/context get more say.

------------------------------------------------------------

"Preference" controls that prevent bad overrides

This is likely what you mean by confidence preferences / preference settings:

1. Symptom floor smoothing (0.10)  
   Prevents one-hot symptom vector from killing all other classes.

2. Geo floor smoothing (0.04)  
   Same idea for over-peaked geo priors.

3. Conflict handling  
   If wound is strong (>= 0.50) and symptom argmax disagrees,
   symptom vector is blended toward uniform (mix = 0.55) to avoid checklist override.

4. Final unknown gate (0.60)  
   Low final confidence -> display unknown.

------------------------------------------------------------

Image quality check

* Uses Laplacian variance sharpness score.
* Very lenient policy.
* Retake recommended only for extreme blur (threshold ~32.0).
* It is a technical blur metric, not clinical wound quality.

------------------------------------------------------------

Species ranking output

snake_species_top is generated by combining:
* current venom distribution,
* region table,
* optional symptom text boosts.

This is a hint list, not species confirmation.

------------------------------------------------------------

If checkpoint is missing (very common question)

If wound checkpoint is absent:
* API still works,
* image branch becomes uniform prior,
* final ranking mostly comes from symptoms + geo + context,
* API returns warning text (fusion_warning).

You can fix it by:
* placing models/wound_ensemble.pt, or
* setting WOUND_ENSEMBLE_URL / WOUND_CHECKPOINT_URL for download on startup.

Viva / interview Q&A (ready to use)

Q1) Why not use only image model?
A: A wound photo may miss systemic signs. Symptoms and context capture clinical clues; geo adds regional priors. Multimodal is more robust than image-only.

Q2) Why log-space fusion?
A: Log-space combines evidence more stably and avoids poor behavior from simple linear averaging when one modality is over-peaked.

Q3) Why give 80% weight to wound when confident?
A: Because a high-quality, confident image model should not be easily overridden by noisy one-hot symptom entries.

Q4) What happens when confidence is low?
A: If final top confidence is below 0.60, UI-facing class becomes unknown to reduce misleading certainty.

Q5) How do you define wound uncertainty?
A: Wound is uncertain if top confidence < 0.65 or top1-top2 margin < 0.12.

Q6) What if model checkpoint is missing in production?
A: System still runs; wound branch becomes uniform prior, and output depends on symptom+geo+context until checkpoint is added.

Q7) Is species output a true identification?
A: No. It is a ranked hint from venom distribution + region + symptom text boost.

Q8) Is geo prior equal to real bite incidence?
A: No. It is occurrence-derived prior, not direct epidemiological incidence data.

Q9) What are the 5 classes?
A: Cytotoxic, hemotoxic, neurotoxic, non_venomous, not_snakebite.

Q10) Is this deployable for hospital decisions today?
A: Not as a certified clinical device. It is educational/research support and should not replace medical judgement.

Q11) Why these exact three backbones and not others?
A: They are reliable and complementary CNN families with stable PyTorch support. The goal is practical robustness and reproducibility on this dataset, not just newest architecture hype.

Q12) Why not use only Vision Transformer?
A: ViT can be strong, but requires dedicated retraining/tuning and careful data-scale matching. For this project phase, ensemble CNNs gave predictable behavior with lower deployment risk.

Q13) Why not just use one giant model and remove symptoms/geo?
A: Snakebite severity patterns are not always visible from one image. Symptoms and region provide important non-image evidence and improve fallback when image confidence is low.

Q14) Why not trust symptom checklist more than image?
A: Checklist input can be noisy or incomplete. The system smooths symptom influence and reduces override risk when image is strongly confident.

60-second presentation script (quick)

"SnakeBiteRx is a multimodal research system for venom-type estimation. It combines wound image CNN output, symptom knowledge-base probabilities, geo priors, and patient context. The image branch uses a 3-model ensemble (EfficientNet-B3, ResNet50, DenseNet121) and uncertainty gates. Final fusion is done in log-space with adaptive weights: image dominates when confident, but symptom and geo gain weight when image is uncertain or missing. The API returns full probability vectors, uncertainty flags, branch-level explanations, and species hints. Safety is built in: low final confidence is shown as unknown. This is educational decision-support, not a medical diagnosis tool."
